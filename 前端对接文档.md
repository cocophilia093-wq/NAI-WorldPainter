# 前端对接文档

## 1. 项目说明

本项目当前是一个本地 Flutter 客户端，核心能力是：

- 用户本地填写 `API Key` 和 `Base URL`
- 调用 NovelAI Gateway 风格接口进行图片生成
- 本地串行队列管理生成任务（同一时间仅 1 个任务执行）
- 本地 SQLite 保存历史记录
- 生成完成后自动下载图片到本地应用目录
- 支持从历史详情页保存到系统相册

当前网络主接口使用：

- `POST /v1/chat/completions`
- `GET /v1/models`

不是服务端中转架构，属于**纯本地前端直连 Gateway**。

---

## 2. Base URL 规则

用户在设置页填写 `Base URL` 时：

- 只填写站点根地址，例如：`https://your-gateway.com`
- 不要手动拼 `/v1`

客户端内部会自动请求：

- `GET {baseUrl}/v1/models`
- `POST {baseUrl}/v1/chat/completions`

如果用户填了 `/v1`，当前代码也会自动去掉尾部 `/v1` 后再拼接接口。

参考实现：`lib/data/datasources/remote/novelai_api_service.dart:25`

## 6. prompt 组织方式

### 6.1 主 prompt

主 prompt 放在 `messages[0]`：

```json
{
  "role": "user",
  "content": "{\"prompt\":\"...\",\"size\":[832,1216],\"negative_prompt\":\"...\"}"
}
```

注意：

- `content` 不是对象，而是 **JSON 字符串**
- `size` 为 `[width, height]`
- `negative_prompt` 仅在非空时传入

参考实现：`lib/data/datasources/remote/novelai_api_service.dart:141`

### 6.2 多角色 prompt

当存在角色列表时，会额外追加一个 `system message`：

```json
{
  "role": "system",
  "content": "Characters: [...]"
}
```

其中 `Characters:` 后面跟的是 JSON 数组字符串，例如：

```json
[
  {
    "prompt": "girl A",
    "center": { "x": 0.3, "y": 0.5 },
    "uc": "bad hand"
  },
  {
    "prompt": "girl B",
    "center": { "x": 0.7, "y": 0.5 }
  }
]
```

角色字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `prompt` | string | 是 | 角色提示词 |
| `center.x` | number | 否 | 横向位置，0~1；不传表示 AI 决定位置 |
| `center.y` | number | 否 | 纵向位置，0~1；不传表示 AI 决定位置 |
| `uc` | string | 否 | 该角色的负面提示词 |

位置规则：

- 手动位置：传 `center`
- AI 决定位置：不传 `center`
- 已隐藏/未启用角色：不进入 `Characters` 数组

完整 5×5 点位表：

| 位置 | A | B | C | D | E |
|---|---|---|---|---|---|
| 1 | `A1` `(0.1, 0.1)` | `B1` `(0.3, 0.1)` | `C1` `(0.5, 0.1)` | `D1` `(0.7, 0.1)` | `E1` `(0.9, 0.1)` |
| 2 | `A2` `(0.1, 0.3)` | `B2` `(0.3, 0.3)` | `C2` `(0.5, 0.3)` | `D2` `(0.7, 0.3)` | `E2` `(0.9, 0.3)` |
| 3 | `A3` `(0.1, 0.5)` | `B3` `(0.3, 0.5)` | `C3` `(0.5, 0.5)` | `D3` `(0.7, 0.5)` | `E3` `(0.9, 0.5)` |
| 4 | `A4` `(0.1, 0.7)` | `B4` `(0.3, 0.7)` | `C4` `(0.5, 0.7)` | `D4` `(0.7, 0.7)` | `E4` `(0.9, 0.7)` |
| 5 | `A5` `(0.1, 0.9)` | `B5` `(0.3, 0.9)` | `C5` `(0.5, 0.9)` | `D5` `(0.7, 0.9)` | `E5` `(0.9, 0.9)` |

通用计算：

```ts
const x = 0.1 + col * 0.2; // col: A=0 ... E=4
const y = 0.1 + row * 0.2; // row: 1=0 ... 5=4
```

示例：

```json
[
  {
    "prompt": "girl A",
    "center": { "x": 0.3, "y": 0.5 },
    "uc": "bad hand"
  },
  {
    "prompt": "girl B"
  }
]
```

其中 `girl B` 未传 `center`，表示由 AI 决定位置。

参考实现：`lib/data/datasources/remote/novelai_api_service.dart:154`

角色实体参考：`lib/domain/entities/generation_task.dart:98`

### 6.3 角色控制草稿保存

角色控制板块需要像主 prompt 一样保存草稿，用户退出页面或重启应用后应恢复当前角色配置。

需要保存的内容：

- 角色数量，最多 6 个
- 每个角色的 `prompt`
- 每个角色的 `uc`
- 每个角色的 `enabled`
- 每个角色的 `centerX/centerY`
- 当 `centerX/centerY` 为 `null` 时表示 AI 决定位置
- 角色顺序

Flutter 当前使用 `SharedPreferences` 保存为 JSON 字符串，key 为：

```text
characters_draft
```

参考实现：

- `lib/core/constants/app_constants.dart`
- `lib/data/datasources/local/settings_local_datasource.dart`
- `lib/presentation/viewmodels/generation_viewmodel.dart`

## 8. 响应解析规则

### 8.1 当前主流程支持的响应格式

代码优先按 Chat Completions 风格解析：

```json
{
  "choices": [
    {
      "message": {
        "content": "![image](https://example.com/xxx.png)"
      }
    }
  ]
}
```

或者多模态：

```json
{
  "choices": [
    {
      "message": {
        "content": [
          {
            "type": "image_url",
            "image_url": {
              "url": "https://example.com/xxx.png"
            }
          }
        ]
      }
    }
  ]
}
```

### 8.2 备用支持

也兼容：

```json
{
  "data": [
    {
      "url": "https://example.com/xxx.png"
    }
  ]
}
```

### 8.3 成功判定

只要能解析出：

- `imageUrl` 或
- `imagePath`

任务就会被标记为 `success`。

否则标记为 `failed`，错误信息为：

```text
未能从响应中提取图片
```

参考实现：`lib/data/datasources/remote/novelai_api_service.dart:173`

## 9. 图片下载行为

接口返回图片 URL 后，客户端会立即二次下载图片到本地：

- 下载仍使用当前已配置的 Dio
- 下载请求会带 `Authorization: Bearer {API_KEY}`
- 下载成功后写入应用文档目录
- 文件名格式：`nai_{timestamp}.png`

本地目录逻辑参考：

- `lib/core/utils/image_utils.dart:11`
- `lib/data/datasources/remote/novelai_api_service.dart:77`

这意味着前端展示时优先使用本地 `imagePath`，其次才是 `imageUrl`。

## 10. 任务状态模型

当前生成任务状态有：

```text
pending
generating
success
failed
```

常量定义：`lib/core/constants/app_constants.dart:31`

### 状态流转

1. 用户点击生成
2. 创建本地任务，状态为 `pending`
3. 入队
4. 队列开始处理后改为 `generating`
5. 接口成功并解析出图后改为 `success`
6. 失败时改为 `failed`

队列实现参考：`lib/core/queue/generation_queue.dart:60`

## 11. 串行队列机制

当前不是并行生成，而是**严格单并发**：

- 同一时间只会处理 1 个任务
- 新任务进入内存队列等待
- 当前任务完成后自动处理下一个

队列状态结构：

```ts
interface QueueState {
  isProcessing: boolean;
  currentTask?: GenerationTask | null;
  pendingCount: number;
  pendingTasks: GenerationTask[];
}
```

实际定义参考：`lib/core/queue/generation_queue.dart:114`

前端若要重做 Web/桌面版，可保留这个模型。

## 17. 目前已知限制

1. 当前主接口依赖 `/v1/chat/completions` 返回图片 URL 或 markdown 图片
2. `b64_json` 虽然做了兼容入口，但还没有走完整的异步落盘逻辑
3. 目前数据库里仍保留 `batch_id` 字段，属于历史遗留
4. 当前是本地单队列，不支持真正并行生成
5. `steps` 目前固定 28，没有开放 UI 修改

## 19. 如果要做 Web 前端，最小接口契约建议

建议你把前端内部统一成下面这套请求结构：

```ts
interface GenerateImageRequest {
  model: string;
  prompt: string;
  negativePrompt?: string;
  width: number;
  height: number;
  scale: number;
  cfgRescale: number;
  sampler: string;
  noiseSchedule: string;
  seed?: number;
  characters?: Array<{
    prompt: string;
    uc?: string;
    centerX: number;
    centerY: number;
  }>;
}
```

转换到实际网关请求时再组装成：

- 顶层参数
- `messages[0].content` 的 JSON 字符串
- `Characters: ...` 的 system message

这样前端内部状态会更干净。
